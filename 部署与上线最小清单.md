# 部署与上线最小清单（Script Generation MVP）

本文档用于把当前后端从“可联调”推进到“可稳定运行”。

## 1. 环境变量清单

必填：

- `DATABASE_URL`
- `SCRIPT_GENERATION_API_KEY`（通用）或 `OPENAI_API_KEY` 或 `GEMINI_API_KEY`

建议填：

- OpenAI：
  - `SCRIPT_GENERATION_PROVIDER=openai`
  - `SCRIPT_GENERATION_MODEL=gpt-4.1-mini`
  - `SCRIPT_GENERATION_BASE_URL=https://api.openai.com/v1`
- Gemini：
  - `SCRIPT_GENERATION_PROVIDER=gemini`
  - `SCRIPT_GENERATION_MODEL=gemini-2.0-flash`
  - `SCRIPT_GENERATION_BASE_URL=https://generativelanguage.googleapis.com/v1beta`
- `SCRIPT_GENERATION_TIMEOUT_MS=30000`
- `SCRIPT_GENERATION_LANGUAGE=zh-CN`
- `PORT=3000`

## 2. 部署前检查（必跑）

```bash
npm ci
npm run prisma:generate
npm run prisma:validate
npm run prisma:migrate:deploy
npm run worker:scripts:check-config
npm run test:integration
```

或直接一键执行：

```bash
npm run release:preflight
```

说明：

- `worker:scripts:check-config` 会只做运行时配置校验，不会消费任务。
- 若 API key 缺失或 provider 配置错误，命令会直接失败，阻止上线。

## 3. 服务启动顺序

1. 启动 API：

```bash
npm run start
```

2. 启动 worker（单次消费模式，适合由 cron/scheduler 周期触发）：

```bash
npm run worker:scripts:once
```

可选限流：

```bash
node src/workers/script-generation-worker.js --limit=5
```

### 3.1 Worker 定时调度接入（基于 `worker:scripts:once`）

推荐使用仓库内置 GitHub Actions 定时任务：

- 工作流文件：`.github/workflows/worker-scripts-scheduler.yml`
- 调度频率：每 2 分钟（`*/2 * * * *`）
- 并发策略：`concurrency.group=worker-scripts-once`，避免重叠消费
- 执行链路：`npm ci -> prisma:generate -> prisma:migrate:deploy -> worker:scripts:check-config -> worker:scripts:once -- --limit=<N>`

上线前需在仓库中配置：

- `Secrets`：`DATABASE_URL`、`SCRIPT_GENERATION_API_KEY`（或 `OPENAI_API_KEY` / `GEMINI_API_KEY`）
- 可选 `Variables`：`SCRIPT_GENERATION_PROVIDER`、`SCRIPT_GENERATION_MODEL`、`SCRIPT_GENERATION_BASE_URL`、`SCRIPT_GENERATION_TIMEOUT_MS`、`SCRIPT_GENERATION_LANGUAGE`

手动补偿跑批：

- Actions 页面手动触发 `Worker Scripts Scheduler`，并按需传入 `limit`

## 4. 冒烟检查（上线后 5 分钟内）

1. 健康检查：`GET /health` 返回 `ok=true`
2. 创建脚本任务：`POST /api/batches/:id/script-generation-tasks` 返回 `202`
3. 轮询批次：`GET /api/batches/:id`
4. 确认任务状态：
   - 正常链路：`QUEUED -> RUNNING -> SUCCEEDED`
   - 非可重试错误：`RUNNING -> FAILED`
   - 可重试错误：`RUNNING -> QUEUED` 且 `retryCount` 增加
5. 确认 `data.scripts` 数量增长

## 5. 运行观察点（日志）

worker 日志为 JSON line，关键字段：

- `event`: `worker_runtime_validated` / `task_succeeded` / `task_failed_retry_scheduled` / `task_failed` / `worker_run_completed`
- `errorCode`
- `alertLevel`
- `metricKey`
- `retryable`
- `errorCodeCounts`

## 6. 常见故障排查

- `SCRIPT_PROVIDER_AUTH`
  - 含义：API key 无效或权限不足
  - 动作：检查 key、项目权限、环境变量注入
  - 重试策略：不可重试（直接失败）

- `SCRIPT_PROVIDER_INVALID_RESPONSE`
  - 含义：上游返回格式不符合预期
  - 动作：检查模型/响应格式、抓请求与响应样例
  - 重试策略：不可重试（直接失败）

- `SCRIPT_PROVIDER_RATE_LIMIT`
  - 含义：限流
  - 动作：降并发、分流、申请配额
  - 重试策略：可重试（回队列）

- `SCRIPT_PROVIDER_TIMEOUT`
  - 含义：请求超时
  - 动作：增大超时、降低单次负载、观察网络质量
  - 重试策略：可重试（回队列）

- `SCRIPT_PROVIDER_NETWORK`
  - 含义：网络连接异常
  - 动作：检查 DNS、出口网络、防火墙
  - 重试策略：可重试（回队列）

- `WORKER_ERROR`
  - 含义：业务前置条件问题或内部错误
  - 动作：看 `errorMessage`，优先补齐批次前置条件（productProfile、selected reference）

## 7. 联调验收与故障演练（429、invalid_response）

建议至少执行：

```bash
npm run test:integration
npm run test:integration:fault-drill
```

通过标准：

- 429 演练：任务状态 `RUNNING -> QUEUED`，`errorCode=SCRIPT_PROVIDER_RATE_LIMIT`，`retryCount` 增加
- invalid_response 演练：任务状态 `RUNNING -> FAILED`，`errorCode=SCRIPT_PROVIDER_INVALID_RESPONSE`，不进入重试

## 8. 回滚建议

触发条件（任一满足）：

- 5 分钟内连续出现 `SCRIPT_PROVIDER_AUTH` 或 `SCRIPT_PROVIDER_CONFIG`
- 任务失败率持续高于 20%
- worker 无法完成 `worker_runtime_validated`

回滚动作：

1. 停止 worker 调度
2. 保持 API 在线（避免前端全不可用）
3. 修复配置后执行 `npm run worker:scripts:check-config`
4. 小流量恢复 worker

## 9. 上线前最后一步（发布闸门）

一键闸门命令：

```bash
npm run release:go-live-gate
```

该命令会顺序执行：

- `release:preflight`
- `test:integration:fault-drill`（429 + invalid_response）
- `GET /health` 冒烟检查

GitHub 最终核对（手工打勾）：

- Secrets 已配置：`DATABASE_URL`
- Secrets 已配置：`SCRIPT_GENERATION_API_KEY`（或 `OPENAI_API_KEY` / `GEMINI_API_KEY`）
- 可选 Variables（未配时走默认）：`SCRIPT_GENERATION_PROVIDER`、`SCRIPT_GENERATION_MODEL`、`SCRIPT_GENERATION_BASE_URL`、`SCRIPT_GENERATION_TIMEOUT_MS`、`SCRIPT_GENERATION_LANGUAGE`
- `CI` workflow 最近一次通过（含 `worker:scripts:check-config`）
- `Worker Scripts Scheduler` 可手动触发成功，日志出现 `worker_run_completed`

可选自动核对（推荐）：

```bash
export GITHUB_TOKEN="<github_pat_with_repo_and_actions_read>"
export GITHUB_REPOSITORY="<owner>/<repo>"
npm run release:github-readiness
```

说明：

- 该命令会检查必需 Secrets、可选 Variables、以及 `CI`/`Worker Scripts Scheduler` 最近一次完成运行状态。
- 若命令失败，按输出提示补齐配置或修复失败工作流后再重试。

## 10. 已知提示说明（DeprecationWarning）

当前依赖组合：

- `@prisma/adapter-pg@7.5.0`
- `@prisma/client@7.5.0`
- `pg@8.16.3`

在高频事务测试下，Node 可能打印如下 deprecation 提示（不影响测试正确性）：

- `Calling client.query() when the client is already executing a query is deprecated...`

处理策略（当前仓库已落地）：

- `test:contract`、`test:integration`、`test:integration:fault-drill` 统一使用 `node --no-deprecation`，避免 CI/本地日志噪音。
- 业务功能与契约断言仍保持完整执行，测试通过标准不变。
- 上游依赖有新版后，再评估是否移除该参数并回归验证。
